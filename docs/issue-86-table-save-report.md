# Issue #86 表格保存问题：分析与解决报告

> 状态：已修复保存链路及后续宽表交互问题。更新时间：2026-07-21。

## 用户现象

在富文本表格中通过行列手柄新增行或列、填写部分单元格后保存，可能出现：

- 输入内容跑到相邻单元格，或多个输入合并；
- 空单元格保存为可见的 `<br />`；
- 连续编辑后凭空多出空行或空列；
- 保存、关闭、重新打开后问题仍存在。

## 根因

这是两层 Markdown 序列化处理叠加造成的低级设计错误，不是表格 UI 或文件保存 IPC 的问题。

1. `markdown-source-preservation.js` 原本把 Crepe 的规范 Markdown 变化按普通文本差分拼回原始 Markdown。表格的 `|`、分隔行和换行不属于可见文字流，连续增删行列时，局部拼接没有可靠的单元格边界，可能把管道或换行放进相邻单元格。
2. Milkdown 在空单元格中会生成 `<br />`，以保证 Markdown 序列化时仍保留该单元格和列数。此前尝试在 remark HTML handler 中提前删除它，但此时表格结构尚未完整输出，而且实际节点常位于 `tableCell → paragraph` 之下；这既没有稳定命中占位节点，也不具备安全删除的时机。
3. 之前的 UI 自动化只检查了保存前页面和落盘内容，没有真正关闭编辑器并从磁盘重新打开文件，因此把“重开后又出现”的路径漏掉了。

## 修复方案

- 只要检测到富文本事务改动了 Markdown 表格，放弃局部 raw-source 拼接，采用 Crepe 生成的完整规范 Markdown。这样表格的管道、行和列由同一份结构化输出一次性决定，不会跨单元格错位。
- 完整 Markdown 输出后，`normalizeEmptyTableCells()` 仅扫描含 GFM 分隔行的真正表格块，将“单元格唯一内容是 `<br />`”规范为 `| |`。`text<br>text` 等真实单元格换行保持不变。
- 删除不可靠的提前 HTML 过滤器；表格内真正的 Enter / Shift+Enter 换行仍由 `tableCellBreakHandler` 输出 `<br>`，再由 remark 插件解析回 hardbreak。

表格结构编辑会因此得到该次文档的规范 Markdown，而不是承诺逐字符保留旧格式。这是明确的安全取舍：宁可规范化，不能写坏用户文件。普通局部文字编辑仍走原文保真映射。

## 回归覆盖

`npm run test:issue-86-ui` 使用真实 Electron 和真实表格手柄，执行：

1. 连续新增两行、两列；
2. 在最后一行五个不同单元格分别输入文本；
3. 保留一处真实的 `text<br>text` 单元格换行，以及一整行未填写的空单元格；
4. 切到源码，检查每个文本仍在原列、表格尺寸为 5×5、空单元格为 `| |`；
5. 返回富文本点击保存，读取真实文件；
6. 彻底退出 Electron，以全新用户目录重新打开同一个文件，再次检查源码和表格尺寸。

同时执行：

```bash
npm run test:markdown-preservation
npm run test:issue-86-ui
npm run build
```

发布前人工按 `docs/manual-test-checklist.md` 的表格项再跑一次，重点确认“保存后关闭并重开”。

## 后续宽表交互补充

后续反馈还覆盖两项与保存无关、但同属 #86 表格体验的问题：

- 表格最右列右键选择列手柄后，表格横向滚动会回到最左端；
- 以前每列至少 `9rem`，普通多列表格即使内容不长也过早需要横向滚动。

根因是右键选择会触发 ProseMirror 选区更新，Crepe 随后刷新表格 node view；其内部 `.table-wrapper.scrollLeft` 不属于编辑器状态，因此重建后回到默认的 `0`。修复在右键菜单打开前捕获当前表格的稳定序号和横向位置；node view 更新后按该序号重新查找新的 wrapper，并连续三个 animation frame 恢复位置，不能再向已分离的旧 DOM 写入。列宽改为内容优先的自动布局，单元格最小宽度降到 `6rem`、最大宽度为 `20rem`；短表保持自身内容宽度，只有内容确实超过编辑区时才保留表格内部横向滚动。

这轮视觉检查还发现了一个主题变量作用域错误：表格底色使用的 `--editor-subtle-surface` 原先放在 `:root`，但它依赖的 `--bg-elevated` 和 `--code-bg` 只定义在 `body.light` / `body.dark`。在实际表格节点计算时该变量会退化为空值，表格底色透明。现已将该变量和其依赖放在同一主题作用域；浅色和深色主题均保留细微表格底色，表头与表体保持可辨识层级。

`npm run test:table-ui` 同时验证桌面和移动窄宽度下：应用页面不横向溢出、短 Markdown 表格不被拉满编辑区且有可见底色、浅色/深色主题中表头与表体可区分、宽 Markdown/HTML 表格可独立横向滚动、最右列右键后 scrollLeft 保持不变，以及新增行列按钮仍完整可点击。

## 列宽与控件补充（2026-07-21）

后续验收发现三个独立的宽表交互问题：拖动列边界只写入内部状态、不改变可见列宽；黄色列边界引导线没有完整贴合表格；横向滚动条与最后一行之间留有不必要的空隙。

### 根因与修复

1. Milkdown GFM 将列宽拖动作为可选插件提供，Crepe 的表格功能没有默认注册它。注册后，插件会把宽度保存到单元格的 `data-colwidth`。
2. Crepe 使用自定义 `TableNodeView`，其表格没有 ProseMirror 默认视图提供的 `colgroup`。因此 `data-colwidth` 虽然正确保存，浏览器没有任何列元素可据此布局，表现为“能拖、但宽度不变”。直接给 `th/td` 写宽度也不正确：它们位于 Crepe 的 `contentDOM` 中，会被 ProseMirror 重新接管并覆盖。
3. 原生插件插入的列边界 widget 会与 Crepe 的加列悬浮控件竞争边缘命中。更关键的是，该插件的实时拖动默认要求表格第一个子节点为 `colgroup`；Crepe 的自定义 `TableNodeView` 以 `tbody` 作为可编辑内容节点，拖动开始时的 node-view 更新会移除临时 `colgroup`，因此此前出现“松手后才变化”。
4. 通用表格底部 `margin` 位于可滚动 wrapper 内，导致 margin 落在最后一行与横向滚动条之间。

修复后的职责边界：`editor-crepe-setup.js` 注册 GFM 列宽插件以提供列边缘识别和引导 decoration；`editor-dom-layout.js` 在长按启动后维护一个不写入文档的 `colgroup` 预览层，鼠标每次移动都直接更新该层，松手时才以一次 ProseMirror 事务写入标准 `data-colwidth`，随后按已提交数据重新同步；表格 CSS 只处理引导线、滚动容器与外部文档间距。这样避开 Crepe node-view 重建与原生拖动 DOM 假设的冲突，黄色列宽引导线保持在真实表格上下边界内，横向滚动条紧贴网格底部。

交互规则也明确分离：普通悬停单元格边界时只显示加行/加列按钮；必须在列边界按住约 220ms 才进入调整模式。进入后引导线为 1px，随实时列宽预览移动；若在等待期间移动超过 4px 或直接松开，则取消调整，不产生文件修改。

### 回归覆盖补充

`npm run test:table-ui` 现额外断言：

1. 普通悬停列边界时仅出现加列按钮；在边界按住约 220ms 后拖动，实际列宽须在鼠标移动的同一过程实时变化，松手后 `data-colwidth` 和实际渲染宽度均增加；
2. 列宽引导线从表格顶部至底部连续对齐；
3. 同一张表的加列、加行按钮仍可被命中并实际改变行列数；
4. 宽表只在自身 wrapper 中滚动，短表保持自然宽度，桌面与移动窄宽度均不撑开应用。

## 防再犯规则

- 不要把结构化 Markdown（表格、列表、标题层级）当作纯可见文本差分处理。
- 不要在序列化中途删除编辑器的结构占位节点；先得到完整输出，再在明确语法边界内规范化。
- 涉及保存的问题，自动化必须至少覆盖一次真实写盘和全新进程重开，不能只断言当前 React/ProseMirror 内存状态。
