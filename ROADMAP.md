# HorseMD 路线图 / Roadmap

> 这里记录 HorseMD 的产品方向与大致优先级，具体顺序会根据稳定性、用户反馈和维护成本调整。
> 想法与问题请提交到 [Issues](https://github.com/BND-1/horseMD/issues)。

## 当前阶段

HorseMD 已具备 Windows、macOS、Android 和 iOS 共用的 Markdown 编辑器内核。当前重点是：

- 保持富文本、源码、大纲、查找和 Review 等核心编辑链路稳定。
- 完善多文件、多根工作区和大文档体验。
- 新功能继续使用聚焦模块，不把逻辑重新堆回 `App.jsx` 或 `Editor.jsx`。
- 桌面与移动端共用 renderer；平台专属能力通过 `window.api.capabilities` 隔离。

## 0.6.0 已完成

- **飞书式斜杠菜单**：支持中文、英文、别名、拼音和拼音首字母匹配；输入语言名可直接创建对应语言的代码块。
- **单一多根工作区**：一个工作区可添加多个文件夹路径，可独立展开、移除和监听，不再用后加入的文件夹替换原目录。
- **大纲默认层级优化**：默认保留前两层实际目录可见；只有一级标题的扁平文档不会被折叠成一个条目。
- **源码/富文本切换稳定性**：强化表格、行内代码、代码块、重复文本和大图文档中的双向光标与阅读位置恢复。
- **源码查找体验**：`Ctrl/Cmd+F` 可跳转、居中并高亮当前匹配；跨富文本/源码切换时重建正确的搜索结果。
- **移动端文本选择**：双击或长按选中文本时只保留系统菜单，避免 HorseMD 工具栏与复制/粘贴菜单重叠。
- **安全与质量基础**：限制外部协议和本地字体权限，主进程按领域拆分，并把核心回归脚本接入 CI。

完整候选更新说明见 [docs/release-v0.6.0.md](./docs/release-v0.6.0.md)。

## 自定义快捷键（第一版已落地）

自定义快捷键来自 [Issue #63](https://github.com/BND-1/horseMD/issues/63)。第一版已经建立统一命令系统，不再向各组件继续追加零散 `keydown` 判断：

1. **统一命令注册表**：为应用命令和首批编辑器块命令定义稳定 ID、默认快捷键、作用域与执行入口。
2. **设置页录制**：直接按键录入组合键，支持清除、单项恢复和全部恢复默认。
3. **冲突与保留键校验**：阻止重复绑定以及覆盖普通文本输入、复制、粘贴、撤销、退出等系统关键组合。
4. **跨平台显示**：同一配置在 macOS 显示 `Cmd`，在 Windows/Linux 显示 `Ctrl`；菜单提示、工具栏和命令面板同步更新。
5. **首批可配置动作**：文件操作、命令面板、PDF、查找/替换、侧边栏、大纲、源码模式、主题、标签切换、Review 命令和标题/正文块切换。
6. **双层分发**：应用级快捷键与 Electron 菜单保持单一触发；设置页显示时阻断后台文档命令，避免保存、查找或侧边栏误触发。

后续仍保守冻结：粗体、斜体、高亮、代码、行内代码、公式、表格结构键、CodeMirror 结构键和输入法相关键。它们直接参与 ProseMirror/CodeMirror 和第三方插件生命周期，后续必须逐项迁移并配套表格、代码块、中文输入法、大文档和模式切换回归，不能只在设置页里先开放。

详细设计与执行入口：

- [设置中心与自定义快捷键架构](./docs/custom-shortcuts-architecture.md)
- [设置中心与自定义快捷键实施清单](./docs/custom-shortcuts-implementation-checklist.md)

> #63 中 H5/H6 强制大写、标题公式转义属于独立编辑器问题，不并入快捷键 Roadmap，应分别修复和测试。

## AI 能力（后期探索）

AI 暂不进入 `0.6.0`，待核心版本发布稳定后再单独设计。当前倾向是：

1. **原生 AI 写作体验**：选区润色、改写、翻译、总结、当前文档问答和工作区上下文由 HorseMD 提供统一 UI。
2. **可插拔 Provider**：通过内部稳定合同适配 OpenAI、Anthropic、OpenAI-compatible 和本地模型，不让上层界面绑定单一厂商协议。
3. **桌面 Agent 模式**：评估 Codex、Claude Code/Agent SDK 等适配器，用于多文档和工作区任务；读取、写入、命令执行必须有明确权限。
4. **Review-first 修改**：AI 不直接静默覆盖正文，优先复用现有 Review/CriticMarkup 展示差异，由用户接受或拒绝。
5. **插件扩展后置**：先稳定 AI session、context、provider、tool 和权限合同，再评估允许插件注册模型、命令、上下文来源和工具；不先建设完整插件市场。
6. **隐私与跨平台**：API Key 不进入 renderer；用户能看到发送了哪些文档；移动端首期只考虑 API 助手，本地 Agent 作为桌面能力隔离。

正式开发前将先完成独立的 AI 产品与架构方案，并用思源、Obsidian AI 插件、Zed Agent 等产品的实际体验校验交互方向。

## 远期方向

- Android Storage Access Framework 外部目录工作区，让用户选择并原地管理设备上的 Markdown 文件（Issue #23）。
- 桌面与移动端文档同步、外部目录镜像/备份，以及 WebDAV 等可自托管存储方案。
- 更完善的导出、发布和跨应用内容工作流。
- 在安全模型与 API 稳定后评估通用插件机制。

## 已完成的主要能力

- 标签式多文件编辑、分屏、文件树、多根工作区、命令面板和会话恢复。
- Milkdown Crepe 富文本与源码模式、查找替换、大纲、Review/CriticMarkup。
- Mermaid、LaTeX、表格、代码块、图片持久化、图床、自定义主题与字体。
- PDF 导出、附件插入、外部文件变更监听和大文件快速打开。
- Windows、macOS、Android、iOS 共用 renderer 与平台能力适配。
